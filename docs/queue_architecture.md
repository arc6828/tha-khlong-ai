# กลไกการทำงานของระบบคิว (FIFO Queue) ร่วมกับ WebSockets

เมื่อระบบถูกปรับเปลี่ยนจากแบบเก่า (ที่นักเรียนเขียนเส้นวาดแย่งกันในหน้าจอเดียว) มาเป็น **"หอศิลป์แยกกระดานวาดภาพส่วนตัวโดยใช้ AI"** ระบบจึงจำเป็นต้องนำกลไก **FIFO Queue (First-In, First-Out)** และ **WebSockets** มาเชื่อมการทำงานร่วมกับ **SQLite** และ **Gemini API** 

หน้าเอกสารนี้จะอธิบายกลไกการทำงานดังกล่าวให้เข้าใจง่ายและเห็นภาพวงจรรูปธรรม

## 1. บทบาทของแต่ละตัวละครในระบบ (Components & Roles)

```
## 3. วัฏจักรชีวิตของบอร์ดผลงาน (Canvas State Transitions)

แต่ละกระดานแคนวาสของการวาดรูป จะมีสถานะเปลี่ยนไปตามการควบคุมดังนี้:

```
[ pending ] ──(เมื่อถึงคิวทำ)──> [ processing ] ──(Gemini ตอบกลับสำเร็จ)──> [ completed ]
 (รอคิวในระบบ)                     (กำลังเรียก Gemini API)               (ประมวลผลเสร็จสิ้น)
                                           └──(Gemini ตอบกลับล้มเหลว)──> [ failed ]
                                                                        (การวาดภาพผิดพลาด)
```

1. **`pending` (รอคิว)**:
   - บอร์ดที่เพิ่งถูกเพิ่มเข้ามาใหม่ทั้งหมดจะเริ่มต้นด้วยสถานะนี้
   - ระบบจะเรียงลำดับพวกมันตาม `createdAt` (เวลาที่กดส่ง) แบบเข้าก่อนออกก่อน (FIFO)
2. **`processing` (กำลังทำงาน)**:
   - ระบบจะหยิบบอร์ดเพนดิ้งใบแรกมาปรับเป็นสถานะนี้
   - ในช่วงนี้ระบบจะ **เรียกใช้ Gemini API** เพื่อแปลงตัวหนังสือเป็นรูปภาพเวกเตอร์
   - ระหว่างที่รอ Gemini ส่งข้อมูลกลับ (ใช้เวลา 2-3 วินาที) แถบความก้าวหน้าบนหน้าจอเด็ก ๆ จะวิ่งนับถอยหลังจำลอง
3. **`completed` (เสร็จสมบูรณ์)** / **`failed` (ผิดพลาด)**:
   - ทันทีที่ Gemini ทำการตอบกลับสำเร็จหรือล้มเหลว ระบบจะปรับสถานะเป็น `completed` หรือ `failed` ทันทีโดยไม่ต้องรอดึงเวลา 5 วินาที
   - หากสำเร็จ: แผ่นแบนพรีวิวภาพจะเปิดขึ้นมาให้เบราว์เซอร์เรนเดอร์ภาพเวกเตอร์ และนักเรียนสามารถกดปุ่มคลิกขยายใหญ่ (Lightbox View) เพื่อเปิดตั้งค่าธีมสีหรือเอฟเฟกต์เฉพาะบอร์ดของตนได้ทันที
   - หากล้มเหลว: หน้าจอจะขึ้นกล่องข้อความแจ้งเตือน "การสร้างภาพล้มเหลว (Gemini API Error)" พร้อมปุ่มให้ลบการ์ดเพื่อยื่นขอวาดใหม่ได้s หน้าจอนักเรียน
    participant NextJS as Next.js API (3000)
    participant DB as SQLite DB
    participant WS as WebSocket Server (3001)
    participant Gemini as Google Gemini API

    Student->>NextJS: 1. ส่งคำขอวาดรูป (action: add_requirement)
    NextJS->>DB: 2. บันทึกคำขอเป็นสถานะ 'pending' พร้อมบันทึกเวลา createdAt
    NextJS->>WS: 3. ยิง HTTP GET /notify (สะกิดฝ่ายหลังร้าน)
    NextJS-->>Student: 4. ตอบกลับผลการรับออเดอร์สำเร็จ
    
    Note over WS: ได้รับสะกิด /notify หรือนาฬิกาครบรอบ 1 วินาที
    WS->>DB: 5. ค้นหาบอร์ดที่ประมวลผลค้างอยู่ และจัดคิวแรกสุดที่ 'pending'
    DB-->>WS: ส่งกลับข้อมูลคิวลำดับถัดไป
    WS->>DB: 6. ปรับสถานะบอร์ดนั้นเป็น 'processing' และเริ่มบันทึก startedAt
    WS->>Gemini: 7. ส่ง Prompt ไปให้ Gemini ออกแบบ SVG (ใช้เวลา 2-3 วินาที)
    Gemini-->>WS: ส่งกลับภาพ SVG (หรือ Error)
    WS->>DB: 8. อัปเดตบันทึกภาพ SVG/Mockup ปรับสถานะเป็น 'completed' หรือ 'failed' และปิด compiles = 0
    WS->>Student: 9. บรอดแคสต์บอกทุกคน (ปิดหน้าจอ Live Coding และแสดงผลภาพทันที)
    WS->>DB: 10. เรียกตัวเองซ้ำเพื่อหยิบคิวถัดไป (Recursive Call)
```

## 3. วัฏจักรชีวิตของบอร์ดผลงาน (Canvas State Transitions)

แต่ละกระดานแคนวาสของการวาดรูป จะมีสถานะเปลี่ยนไปตามการควบคุมดังนี้:

```
[ pending ] ──(เมื่อถึงคิวทำ)──> [ processing ] ──(Gemini ตอบกลับ)──> [ completed / failed ]
 (รอคิวในระบบ)                     (เรียก Gemini API                    (ประมวลผลเสร็จสิ้น
                                และคอมไพล์ความก้าวหน้า)               นักเรียนเปิดชมภาพได้)
```

1. **`pending` (รอคิว)**:
   - บอร์ดที่เพิ่งถูกเพิ่มเข้ามาใหม่ทั้งหมดจะเริ่มต้นด้วยสถานะนี้
   - ระบบจะเรียงลำดับพวกมันตาม `createdAt` (เวลาที่กดส่ง) แบบเข้าก่อนออกก่อน (FIFO)
2. **`processing` (กำลังทำงาน)**:
   - ระบบจะหยิบบอร์ดเพนดิ้งใบแรกมาปรับเป็นสถานะนี้
   - ในช่วงนี้ระบบจะ **เรียกใช้ Gemini API** เพื่อแปลงตัวหนังสือเป็นรูปภาพเวกเตอร์ เมื่อได้รูปภาพแล้วจะนำมาบันทึกเก็บไว้ทันที
   - แถบความก้าวหน้าบนหน้าจอเด็ก ๆ จะวิ่งนับถอยหลังโดยคำนวณจากตัวนับเวลา 5 วินาที (และหน้าจอจะสลับเข้าแสดงผลภาพทันทีที่ Gemini ส่งคำตอบกลับมา)
3. **`completed` (เสร็จสมบูรณ์) / `failed` (ผิดพลาด)**:
   - ทันทีที่ Gemini API ตอบกลับสำเร็จหรือล้มเหลว ระบบจะทำการปรับสถานะเป็นสำเร็จ (`completed`) หรือ ผิดพลาด (`failed`) ในทันที
   - แผ่นแบนพรีวิวภาพจะเปิดขึ้นมาให้เบราว์เซอร์เรนเดอร์ภาพเวกเตอร์ (หรือแสดงข้อผิดพลาดถ้าสร้างไม่สำเร็จ) และนักเรียนสามารถจัดการภาพหรือการ์ดของตนได้ทันที

## 4. ทำไมออกแบบแบบนี้ถึงมีประโยชน์สูงและทำงานได้แม่นยำ?

1. **ไม่เกิด Gemini API Rate-Limit 429**:
   - เนื่องจากเราบังคับให้การดึงคิวเป็น **Single Coordinator** ในหน่วยความจำของฝั่ง WebSocket Server เท่านั้น โดยรัน `await generateCanvasSvg` ทีละอัน และหยุดรอผลลัพธ์แบบ Synchronous ก่อนเลื่อนคิวถัดไป ทำให้ Gemini API จะโดนเรียกทีละงาน ไม่มียิงชนกันขนาน
2. **ประหยัดทรัพยากรการส่งข้อมูลสูงมาก**:
   - การเชื่อมต่อ WebSockets จะลดความถี่การยิง HTTP Request ของเครื่องไคลเอนต์ลง หน้าเว็บของเด็ก 40 คนจะเงียบสนิท ไม่มีการส่งสัญญาณรบกวนเน็ตเวิร์กเลยจนกระทั่งสถานะคิวใน SQLite ขยับจริง ๆ ตัว WebSocket Server จึงจะยิงบรอดแคสต์ข้อมูลอัปเดตไปหาทุกคนในครั้งเดียว
3. **ป้องกันระบบแย่งกันเขียนทับ (Concurrency Safety)**:
   - ด้วยการเปลี่ยนการเขียนข้อมูลไปเป็นคำสั่ง SQL เจาะจงคอลัมน์ (เช่น `UPDATE canvases SET theme = ? ... WHERE id = ?`) แทนที่จะดาวน์โหลดไฟล์ state มาแก้ไขในแรมแล้วเซฟทับตัวใหญ่ ทำให้กระบวนการกดแต่งสไตล์ภาพเวกเตอร์บนหน้าจอของนักเรียน ไม่มีโอกาสไปเบียดเบียนหรือยกเลิกการเขียนความก้าวหน้าการคอมไพล์คิวของตัวหลังร้าน
